#### Anti-Patterns Verdict

**Borderline, leans no.** No gradient text, no glassmorphism, no hero-metric splash — genuine restraint. But two independent checks (an LLM review that read the live page fresh, and the deterministic `impeccable detect` scanner) **both separately flagged the same thing**: `.notice { border-left: 4px solid #d8a900; }` (styles.css:183-193) is the textbook side-stripe accent-border tell. That convergence is a strong signal, not a coincidence.

**Deterministic scan** (`npx impeccable detect index.html`) found 5 issues the LLM review didn't fully cover on the type side:
- Side-tab accent border (confirms the LLM finding above).
- Roboto flagged as an overused/generic font.
- Single font family for the entire page, no pairing.
- Type scale technically spans 12px→22.4px (1.9:1) but most steps are only 0.8px apart — reads as flat in practice despite the endpoint ratio looking fine on paper.
- 27 em-dashes in body copy (advisory) — a known AI-cadence tell, worth a copy pass regardless of intent.

No live browser-overlay pass was run this time — the `impeccable live` command isn't exposed in the currently installed CLI version (only `detect`, `ignores`, `install` etc. are). The independent LLM assessment already inspected the real rendered page at both a desktop (1400×1000) and mobile (390×844) viewport directly, which covers most of what the overlay would have added.

#### Overall Impression

This is a well-disciplined utilitarian tool that mostly avoids the generic-SaaS trap it's explicitly trying to avoid — the color-coded confidence system and the constant self-aware "this could be wrong, here's why" copy are real, non-generic design decisions. The gap is that **writes feel like they vanish into a void**: every add/pause/remove/edit says "leva ~1 minuto, depois clique em Atualizar lista" with no follow-up confirmation, which is the single biggest thing undermining "confiável" for a solo user anxiously checking whether their action landed. Mobile support is real but reactive (one `@media` query, desktop-sized tap targets shrunk down) rather than designed-for, despite mobile being explicitly named as a primary use context in PRODUCT.md.

#### What's Working

1. **Consistent color-coded confidence system** (green=good, yellow=caution, red=avoid, gray=insufficient) reused identically across Flight Score, recommendation, dashboard health, and promotion ranking — this alone does most of the "confiança nos dados" work PRODUCT.md asks for.
2. **Self-aware, honest copy about data limits** — Flight Score/recommendation disclaimers, "amostra curta" notices, the intelligence assistant listing its own limitations. This directly serves the "Honestidade sobre incerteza" principle and is the strongest asset in the whole interface.
3. **The flexible-search preview → count → explicit authorization gate** (invalidated on any field change) is a real, domain-specific error-prevention mechanism, not boilerplate form validation.

#### Priority Issues

- **[P1] No confirmation loop after write actions.** Add/pause/remove/edit all end in "aguarde ~1 minuto, depois clique em Atualizar lista" (app.js, `addFlight`/`changeFlightStatus`/`removeFlight`) with zero automatic follow-up. For a solo user checking whether their edit actually landed, this is the main trust gap in the app. **Fix**: after dispatch, poll `config/flights.json` a few times (e.g. every 15s for up to ~90s) and auto-refresh the list when it changes, or at minimum auto-trigger one refresh after a fixed delay instead of relying on the user to remember. Suggested command: `/impeccable harden`.

- **[P1] Touch targets sized for desktop, shrunk for mobile, not designed for it.** `button { padding: 0.5rem 1rem }` and `input, select { padding: 0.4rem }` are the same values at every breakpoint; the "Pausar"/"Remover" buttons sit close together per flight row with no explicit minimum height. Given PRODUCT.md names mobile as an explicit real usage context (not a fallback), a mis-tap on "Remover" instead of "Pausar" is a real, avoidable risk. **Fix**: deliberate ~44px minimum touch height on interactive elements at the `max-width: 520px` breakpoint, with visible spacing between adjacent destructive/non-destructive actions. Suggested command: `/impeccable adapt`.

- **[P2] Native `window.confirm()` on the one destructive action.** Removing a monitored flight (`removeFlight`, app.js) drops into a browser-native dialog, breaking the app's own visual language at exactly the moment that should feel most considered. **Fix**: inline confirm affordance (button becomes "Confirmar remoção?" on first click) instead of the native dialog. Suggested command: `/impeccable clarify`.

- **[P2] Raw HTTP/JSON errors leak through the fallback path.** `responseError()` (app.js) surfaces `HTTP 401: {"error":"..."}`-style text verbatim, truncated at 1000 characters, which contradicts the otherwise careful, humane error copy everywhere else in the app. **Fix**: map known status codes (401/403/404/dispatch failures) to the same plain-language tone already used for session errors; keep the raw body only behind a collapsed "detalhes técnicos" disclosure. Suggested command: `/impeccable clarify`.

- **[P2] Dashboard has no progressive disclosure.** All sub-panels (promoções, menores preços, maiores quedas/altas, médias por destino/companhia/mês, alertas, falhas) render open simultaneously with no collapse — a lot to scan on both desktop and (especially) mobile. **Fix**: collapsible sections, or a small set of tabs, so the density is revealed on demand rather than all at once. Suggested command: `/impeccable layout`.

- **[P3] Single overused font, flat hierarchy at the low end of the scale, and a side-stripe accent border on `.notice`.** Cosmetic, but a cheap, high-leverage fix given the explicit "nada de SaaS genérico" anti-reference: swap `.notice`'s left border for a full border or background tint, and pick a less-ubiquitous font pairing (or lean harder into weight/size contrast if staying single-family). Suggested command: `/impeccable typeset`.

#### Persona Red Flags

**Alex (Power User)**: No keyboard shortcuts anywhere. No bulk pause/resume/remove across multiple flights — every action is one-at-a-time with a manual follow-up refresh. Would find the ~1-minute wait-then-manually-refresh loop for every single edit genuinely tedious on a day with several changes.

**Casey (Distracted Mobile User)**: This is the closest match to the actual real-world user described in PRODUCT.md. Primary actions (Pausar/Remover) sit mid-page, not in an obvious thumb zone, and are the same shrunk-desktop size noted above — a one-handed, half-attentive tap is genuinely more likely to mis-hit here than on desktop. No state is lost on interruption (nothing is multi-step/session-based beyond the login itself), which is a point in its favor.

**Sam (Accessibility-Dependent User)**: The color-coded confidence system (green/yellow/red/gray) is meaning-by-color in several places (Flight Score card, recommendation card, dashboard health card) — worth checking each also carries a text label, not just a class name driving a background color, since that's the one accessibility risk pattern worth a deliberate look here.

#### Minor Observations

- 27 em-dashes across the visible copy (advisory-level detector finding) — worth a pass with commas/colons/periods per the project's own writing conventions, independent of any design concern.
- The "fixed mode add/update" path silently overwrites an existing flight id with no diff/warning if the id already exists — low-severity but worth a one-line confirmation copy change.
